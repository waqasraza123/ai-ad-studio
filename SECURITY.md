
Security Policy
Supported reporting path

Please do not report security vulnerabilities in public issues, pull requests, or discussions.

Use one of these routes:

GitHub private vulnerability reporting, if enabled for this repository

A private maintainer contact path available from the repository owner profile

What to include

A useful report should include:

affected area or component

reproduction steps

expected impact

environment details

screenshots, logs, or proof of concept when safe to share

whether the issue is publicly known

What happens next

The maintainer will review the report, validate the issue, and decide the next steps for remediation and disclosure.

Typical handling goals:

acknowledge receipt as soon as practical

assess severity and scope

prepare a fix or mitigation

coordinate disclosure responsibly

Please avoid

posting exploit details publicly before a fix exists

opening duplicate public issues for a vulnerability

accessing data that does not belong to you

attempting destructive testing against live systems

Non-security bugs

For ordinary product bugs or feature requests, use the issue templates instead.
